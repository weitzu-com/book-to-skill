# 06 · 治理：三省六部 + 中华经典三层治理

> 本文是 SKILL.md「书道执行本身用三省六部治理」的扩展位。
> 解决一个工程问题：**把一本书做成 skill 是多步、多 Agent、易出错的流水线，
> 谁来起草、谁来审驳、谁来执行？** 答案：用「起草—审驳—执行」的权力分立来防错。
> 用法：每次书道执行前读「治理流程」对齐角色与时序；产物分派看「六部映射表」；
> 方法论卡住时回到「中华经典三层治理」找根。

---

## 1. 为什么要治理（第一性原理）

把书做成 skill，不是一个人一口气写完，而是一条流水线：预处理 → 多 Agent 并行解析 →
认知架构设计 → 编写 → 验证 → 归档。这条链有两个固有风险：

1. **单点起草易盲**：同一个人既起草又自验，看不见自己的盲区——漏概念、孤立节点、
   路由「写了但调不到」，自己往往发现不了。
2. **多 Agent 易散**：并行解析的产物口径不一、边界重叠、隐含假设互相矛盾，
   没有统一执行总管会拼不成一个连贯的 skill。

因此书道引入**权力分立**：把「起草蓝图」「独立审驳」「落地执行」三种权力分给三个角色，
**审驳者必须独立于起草者，并有权驳回**。这就是三省的来历。
对应书道第一性原理：**不以「记住了多少」为验收，以「能否在新场景做出符合原书精神的判断」为验收**——
而判断的质量，靠独立审驳来守门。

---

## 2. 三省映射（决策 / 审查层）

三省管「要不要做、做得对不对」，不直接生产文件。

### 中书省（草诏）= 蓝图与架构设计
- 职责：在动手前定下 skill 的**认知架构五要素**（身份 / 知识分配 / 能力边界 / 行为模式 / 交互协议）、
  目录骨架（references 六件套 + scripts + templates）、以及 **SKILL.md 草案**。
- 产出：一份「书道蓝图」——这本书以谁的身份说话、章节数→Agent 数、六件套各装什么、命令路由怎么设计。
- 原则：草诏阶段只画图不施工；蓝图不清晰就不放行到执行层。

### 门下省（审驳 / 谏省）= 独立评审，**有权驳回**
- 职责：对尚书省的产物做独立验收，是整条流水线的**质量闸门**。核对清单：
  - **五维验证**：概念完整 / 关系完整 / 应用正确 / 判断准确 / 边界清晰（详见 `references/04-validation.md`）。
  - **references 六件套齐全**：concepts / frameworks / methods / principles / cases / knowledge_graph 缺一不可。
  - **路由加载指令完整**：SKILL.md 路由表每条都带 `方法→references/methods.md#方法N` 形式的具体加载指令。
    「写了但调不到 = 知识不存在」。
  - **SKILL.md < 500 行**：超行即驳回拆分到 references。
- 裁决：不合格 → **「驳回」** 标明缺陷退回回炉；合格 → **「通过颁行」**。
- 铁律：门下省不能是中书省/尚书省同一个 Agent 自己——**自审无效**，必须独立视角复核。

### 尚书省（执行总管）= 统辖六部
- 职责：领中书省蓝图，**把蓝图拆成任务分派给六部并行执行**，再把六部产物拼装成一个连贯 skill，
  提交门下省审驳。
- 产出：一棵完整的 skill 目录树（SKILL.md + references/ + scripts/ + templates/ + README + LICENSE）。
- 原则：尚书省只管「落地与协调」，不自己改蓝图（改蓝图要回中书省）、不自己签收（签收归门下省）。

---

## 3. 六部映射（执行层）

尚书省统辖六部，把书道各产物分派下去。**口诀**：吏管身份、户管知识、礼管门面、兵管工具、刑管法度、工管营造。

| 六部 | 掌 | 书道产物 |
|------|------|---------|
| **吏部**（人事/身份）| 官制与任用 | 认知架构「身份定位」+ SKILL.md frontmatter / 命令路由 + 本治理文档与 `templates/` 模板 |
| **户部**（普查/知识）| 户籍与钱粮 | 阶段二**四层解析**（概念→框架→方法→原则）+ references 的 `concepts/frameworks/methods/principles` |
| **礼部**（礼制/门面）| 礼仪与外交 | `README` + `LICENSE` + 对外文档与流程总纲 |
| **兵部**（兵器/工具）| 军备与征伐 | `scripts/` 脚本 + Workflow 并行作战（`references/05-workflow.md`）|
| **刑部**（法度/质检）| 刑律与稽核 | 五维验证 + 防错清单 + `scripts/check_skill.py` 验证 |
| **工部**（营造/工程）| 营建与制造 | 预处理与结构化工程（`references/01-preprocess.md`）+ `templates/` 构造 |

> 注：吏部、工部都触及 `templates/`——吏部定模板的**官制规范**（占位符、frontmatter、路由格式），
> 工部负责把模板**构造落地**到具体 skill。二者一立法、一施工，不冲突。

## 4. 一次书道执行的治理流程（时序）

```
① 中书起草
   读源材料 → 定认知架构五要素 + 目录骨架 + SKILL.md 草案 →「书道蓝图」

② 尚书分派（六部并行）
   户部：四层解析 → 六件套知识体
   工部：预处理/结构化 + 套 templates 骨架
   兵部：scripts + Workflow 并行
   吏部：身份定位 + frontmatter + 命令路由
   礼部：README + LICENSE + 流程总纲
   刑部：边写边查防错清单
   → 拼装成完整 skill 目录树

③ 门下审驳（质量闸门）
   跑五维验证 + check_skill.py + 六件套齐全 + 路由加载指令 + <500行
   ├── 不合格 → 驳回：标明缺陷 → 回炉对应六部重做 → 重新提交 ②/③
   └── 合格 → 通过颁行 ↓

④ 颁行（落地闭环）
   安装到 ~/.claude/skills/  +  推 GitHub  +  原点 KB 归档（知识图谱 + 工作日志 + 索引）
```

**回炉路径要短**：门下省驳回时直接点名「哪一部、哪个文件、哪条标准没过」，
让尚书省精准回炉，不要整盘推翻。

## 5. 中华经典三层治理（方法论根）

三省六部管「权力怎么分」，中华经典三层管「判断从哪来」。二者是书道治理的两条腿。

- **谋（孙子）→ 阶段零 / 三：先识「势」再动手**。
  阶段零预处理决策、阶段三认知架构设计都属「战略设计」——
  先判断这本书的「势」（信息密度、可迁移性、以谁身份说话），再决定打不打、怎么打。
  「不可跳过亲自交叉验证」就是孙子「知彼知己」的落地。

- **道（老子）→ 阶段二 / 六：关系网 > 列表，知识自进化闭环**。
  阶段二解析追求**概念关系网**而非概念列表（孤立节点 = 死知识）；
  阶段六归档建知识图谱完整度热图、回流原点 KB，让 skill 能被后续执行复用与自我进化。

- **行（论语）→ 阶段四 / 五：以用为验，不以记为验**。
  阶段四编写、阶段五五维验证都以「能否在新场景做出符合原书精神的判断」为准绳，
  角色扮演、变体场景、边界场景——验的是「用」，不是「背」。

> 三省六部 × 中华经典三层 = 书道治理双螺旋：
> **权力分立防单点之盲，经典三层定判断之根。**
