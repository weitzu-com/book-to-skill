# 01 · 阶段零/一：格式判断与预处理工程（扫描版 PDF 专项）

> 覆盖 SKILL.md 阶段零（预处理决策 + 冲突检测）与阶段一（结构化）的工程细节。
> 核心命题：**先判格式，再动手**；尤其扫描版 PDF——文件名标 `(OCR)` 不代表能提取文字。

---

## 1. 预处理决策树（判格式 → 选路径 → 估时）

```
源文件是什么格式？
├── MD（已优化）        → 跳过预处理，只查目录/层级完整性              ~2min
├── MD（未优化）        → 全角→半角 + 标题层级规范化（# ## ###）       ~5min
├── PDF（文本型）       → pdftotext -layout → TXT → 全角→半角 → 识别章节 ~10-15min
│                         （scripts/pdf_to_text.sh 一步到位）
└── PDF（扫描型/OCR）   → ⚠️ 必须先验证文字层，见 §2                   视质量而定
```

**文本型 vs 扫描型怎么判**：
1. 用 Read 工具/`pdftotext` 试提一两页。
2. 出来是正常文字 → 文本型。
3. 输出为空 / 全是页码 / 乱码方块 → 扫描型（哪怕文件名带 `(OCR)`）。

---

## 2. 扫描版 PDF 专项

> ⚠️ **第一性铁律**：文件名标 `(OCR)` ≠ Read 工具能提取文字。
> 很多扫描书做过 OCR 但文字层没嵌入 PDF，或嵌入了但 Read 读不到。
> **一律按扫描版处理，先验证文字层**，验证通过才降级当文本型。

### 方案 A（首选）：PyMuPDF 提取嵌入 OCR 文字层
- 脚本：`scripts/extract_pdf.py book.pdf > full_text.txt`
- 原理：用 `fitz`（PyMuPDF）读 PDF 内嵌的文字层（含已 OCR 文字）。
- 性能基准：**258 页 / 15.9MB / <30s**。
- 适用：PDF 已含文字层（绝大多数标 (OCR) 的书属此类）。

### 方案 B（兜底）：tesseract `chi_sim` OCR
- 适用：方案 A 提取为空（PDF 无文字层，纯图片扫描）。
- 依赖：`chi_sim` 语言包约 **42MB，须 `brew` 预装**（`brew install tesseract tesseract-lang`）。
- 性能：逐页图像识别，**1–3 分钟**（视页数）。
- 命令骨架：`pdf 转图片 → tesseract -l chi_sim`。

### 保守修复原则（OCR 文本清理三档）
| 情形 | 处理 |
|------|------|
| 明确可修（如显然的形近误识） | 直接修 |
| 存疑（拿不准是否误识） | 标记 `[OCR可疑]`，留给阶段二 Agent 判断 |
| 完全不可读（整段糊掉） | 标记 `[OCR不可读]`，不强行猜 |

> 原则：**宁可标记，不可臆造**。臆造会污染知识，标记可被下游 Agent 智能补救。

### OCR 质量门禁（进阶段二的闸门）
- **可识别率 ≥60%**：关键语义段落（目录/前言/核心章首段）可读率 ≥60% → 放行进阶段二。
  - 理由：阶段二 Agent 能智能纠正残余乱码，例：`Ｃｔａｕｄｅ→Claude`、`Ｈｏｋｓ→Hooks`。
- **触发重扫**：目录 / 前言 / 核心章首段**完全不可识别** → 不放行，回方案 B 或换更优源（§6）。

## 3. 中文扫描版注意

- **竖排文字被拆成单字行**：OCR 把竖排版逐字拆行。
  - 阶段一**不修复**（不强行合并），原样保留，留给阶段二 Agent 结合语义理解。
- **全角字符混用**：中文扫描常出现全角字母/数字混入正文。
  - **优先 Agent 智能识别**，而非脚本暴力转换——暴力转换会误伤本应全角的内容。
  - 脚本只处理标点/纯英文段的明确全角→半角；正文混排交 Agent。

## 4. 章节边界检测

**检测模式**（搜索这些信号）：
- `第X章` / `第X节` / `第X篇`
- `Chapter N` / `Section N`
- 编号标题（`1.` `1.1` `一、`）+ 页码模式

**扫描版的麻烦**：标题 OCR 变形，例 `登高望远 → 登高望 远`（中间多空格/缺字）。
- 对策：**宽匹配 + 多候选行 + 人工确认**。
  1. 宽匹配（容忍空格/形近字）拉出候选行。
  2. 列出多个候选供核对。
  3. 人工确认后，把**精确行号**写入结构骨架。
- 工具：`scripts/detect_structure.py full_text.txt` 出候选 + 骨架，再人工定行号。

## 5. 冲突检测（动手前必做）

```
搜 ~/.claude/skills/ 是否有同名/近似 skill
├── 已存在            → 问用户「更新 / 新建 / 取消」
├── 已存在且质量高    → 仅补路由加载指令，不推翻已有成果
└── 不存在            → 新建
```

> 近似判定：不只比对目录名，也看 description 主题是否撞车（同一本书/同一方法论）。

## 6. 更优源文件搜索优先级（避免从劣质源苦干）

| 优先级 | 源 | 理由 |
|--------|----|------|
| ① | 优化过的 MD | 已结构化，最省事（~2min 进阶段一） |
| ② | 学习成长目录的 md/txt/pdf/epub | 可能有更干净的版本 |
| ③ | 已有 book-* skill 的预处理产物 | 复用 `full_text.txt` / `book_structured.md`，免重 OCR |

> 工程要点：**扫描版 PDF 苦干前，先花 1 分钟搜更优源**——找到优化 MD 可省下整个 OCR + 清理环节。

> 判错格式（扫描当文本）、漏冲突检测、忽略更优源，任一失误都会让后六个阶段在错误地基上空转。
