# 阶段二 · 多 Agent 并行深度解析

> **一句话**：把一本书「普查」成结构化知识——用多个 Agent 并行把全书拆成
> 概念/框架/方法/原则四层，每个知识要素都钉上书中出处锚点，再由你亲自交叉验证查漏。
>
> **第一性原理**：解析不是「读懂」，是「让书的判断力可被检索、可被复用」。
> 验收标准不是「Agent 读完了」，而是「四层知识齐全 + 每条都能溯源 + 你抽查无遗漏」。

**何时检索本文**：进入阶段二、要派发解析 Agent、要写 Agent prompt、
不确定派几个 Agent、Agent 输出后要交叉验证、或需要提取隐含假设/调和矛盾时。

---

## 1. 四层提取体系（提取维度，正交于五维验证）

四层是**从书里抽什么**的维度；五维验证（见 `04-validation.md`）是**抽完怎么查**的维度。
两者正交：四层管「有没有」，五维管「对不对、调不调得到」。

| 层 | 名称 | 识别信号 | 必含字段 | 输出锚点 |
|----|------|---------|---------|---------|
| L1 | 概念 | 「X 是 Y」「所谓 X，即…」的定义句 | 概念名 + 一句定义 + 与近义概念的区别 | 书中页码/章节出处 |
| L2 | 框架 | 编号列表、步骤体系、分类法、N 要素/N 阶段 | 框架名 + 组成项 + 项间关系 | 章节对照（框架出自哪几章）|
| L3 | 方法 | 「如何做…」「步骤：1.2.3.」可操作段落 | 步骤序列 + 输入 + 输出 + 适用场景 + 验证标准 | 书中操作示范/案例位置 |
| L4 | 原则 | 「应当…」「永远不要…」底层信念、价值判断 | 原则陈述 + 边界条件 + 与其他原则的冲突调和 | 作者反复强调处/反例处 |

**案例不是独立层**。案例是上述每一个知识要素的「验证锚点」：
- 它证明这条概念/方法/原则在书里**真实存在且作者认可**，不是 Agent 脑补。
- 每个知识要素**必须**标注至少一个书中案例或出处（页码/章节）。无锚点的要素一律视为「待核实」，
  在交叉验证阶段优先复查。
- 锚点格式建议：`（第N章 / pXX）` 或 `（案例：客户XX的故事，第N章）`。

> 原则：**没有出处的知识不进库**。一本书做成 skill 后若被质疑「书里真这么说？」，
> 锚点就是答案。

## 2. Agent 数量决策表

按**章节数**决定派几个 Agent。少派 → 漏；多派 → 重复 + 难合并。

| 章节数 | Agent 数 | 分配策略 | 预计耗时 |
|--------|---------|---------|---------|
| < 8 | 1 | 单 Agent 全量通读 | 5–10 min |
| 8–15 | 2 | 前半（基础概念+框架）+ 后半（方法+原则）| 10–15 min |
| 15–25 | 3 | 核心章 A + 后期章 B + 中间补全 C（标准分工，见 §3）| 15–25 min |
| 25+ | 4+ 或 Workflow | 按主题分组并行（pipeline 先到先出，见 `05-workflow.md`）| 20–35 min |

**判断口诀**：宁可让 C 这种「补全 Agent」有重叠，也不要留章节无人覆盖。重叠可在合并时去重，遗漏在交叉验证时才发现，代价更高。

---

## 3. 三 Agent 标准分工模板（15–25 章场景）

这是验证过最稳的分工。核心思路：**A/B 啃两端核心章，C 兜底中间**，B 额外背负「特命任务」。

| Agent | 覆盖范围 | 主任务 | 特命任务 |
|-------|---------|--------|---------|
| A | 前半核心章（1–7）| 作者故事 + 基础概念（L1）+ 核心框架（L2）| —— |
| B | 后半核心章（12–20）| 方法（L3）+ 原则（L4）| **识别隐含假设 + 发现矛盾并调和**（见 §6）|
| C | 中间补全章 | 覆盖 A/B 未覆盖的所有章节，查漏补缺 | 标记与 A/B 重叠/冲突处 |

**为什么 B 背特命**：方法与原则集中在书的后半，隐含假设和内部矛盾也多在此暴露
（作者讲完方法后才显出价值排序）。让啃后半的 Agent 顺手做，比单设一个 Agent 更省。

**章节号是示意**：实际按书的「核心章在哪」滑动。先快速扫目录定位「最厚/最常被引用」的章 → 给 A/B，剩下给 C。

## 4. Agent Prompt 必含四要素

派出去的每个解析 Agent，prompt 必须含以下四块，缺一块就会拿回「散文式读后感」而非结构化知识：

| 要素 | 作用 | 怎么写 |
|------|------|--------|
| ① 输出格式强制 | 逼出四层结构化，便于机器合并 | 指定四层 JSON Schema（StructuredOutput 最佳），或严格 Markdown 表格 |
| ② 识别信号提示 | 让 Agent 知道「看到什么算什么层」| 「X 是 Y」→L1；编号列表→L2；「如何做」步骤→L3；「应当/永不」→L4 |
| ③ 验证锚点要求 | 杜绝脑补，强制溯源 | 每个知识要素**必须**附书中案例/页码/章节出处 |
| ④ 特命任务（如有）| 提取隐性知识 | 隐含假设提取 + 矛盾调和（仅派给承担特命的 Agent，如 B）|

### 可直接复制的 Agent Prompt 范例

```
你是《<书名>》第 12–20 章的深度解析 Agent。请只读这些章节，输出结构化知识。

【输出格式】严格按以下四层输出，用 JSON（StructuredOutput）：
{
  "concepts":   [{"name","definition","distinction","source"}],   // L1
  "frameworks": [{"name","components":[],"relations","chapters"}], // L2
  "methods":    [{"name","steps":[],"input","output","when_to_use","success_criteria","source"}], // L3
  "principles": [{"statement","boundary","conflicts_with","resolution","source"}], // L4
  "hidden_assumptions": [{"assumption","why_implicit","evidence"}], // 特命
  "conflicts":  [{"a","b","context","resolution"}]                  // 特命：矛盾调和
}

【识别信号】
- 「X 是 Y / 所谓 X」定义句 → concepts(L1)
- 编号列表 / N 步骤 / N 要素 / 分类法 → frameworks(L2)
- 「如何做」「步骤 1.2.3」可操作段落 → methods(L3)
- 「应当 / 永远不要 / 必须」价值判断 → principles(L4)

【验证锚点】每一条 concepts/methods/principles 都必须填 source（章节+页码或案例名）。
无法溯源的，标 source="待核实" 并单列。

【特命任务】
1. 隐含假设：找出作者默认成立、但从未明说的前提（如「读者已有客户基础」）。
2. 矛盾调和：若不同章节的原则冲突，记录冲突 + 各自适用边界，不要替作者二选一。

只输出 JSON，不要寒暄。
```

## 5. 亲自交叉验证（不可跳过）

Agent 会漏。漏的往往是「太显然以至于没被当成定义句」的核心概念。**这一步不可省**。

1. **选样**：挑全书 2–3 个知名度最高、最常被外界引用的核心章（不是最长的章，是最「招牌」的章）。
2. **亲自读**：你本人通读这 2–3 章，边读边列「我认为这章必含的概念/框架/方法/原则」。
3. **对照**：拿你的清单逐条比对 Agent 输出。
   - Agent 有、你没有 → 你漏了，正常。
   - 你有、Agent 没有 → **Agent 遗漏关键知识，手工补全**，并反查同类遗漏是否在其他章也发生。
4. **补全后回写**：补的条目同样要钉锚点，进入对应层。

> 经验：招牌章的核心概念若被漏，做出来的 skill 会在最该出彩的地方哑火。交叉验证的 10 分钟，省的是
> 阶段五推翻重做的几小时。

## 6. 隐含假设与矛盾调和

这是把书从「知识清单」升级为「判断力」的关键两步，必须显式做。

### 6.1 为什么要提取隐含假设
作者写书时默认读者共享某些前提，于是从不写出来。但 skill 的使用者**没读过书**，
这些前提一旦缺失，skill 会在不适用的场景里给出错误自信的建议。

- 典型隐含假设：「读者已有稳定客源」「方法适用于 B2B 非 B2C」「数据量足够大」「团队 ≥ 10 人」。
- 做法：让特命 Agent 专门追问「这条方法要成立，作者默认了什么？」，把答案写进
  `knowledge_graph.md` 的隐含假设区，并在 `principles.md` 的相关原则下标注边界条件。

### 6.2 书内原则冲突如何调和
一本厚书里，不同章节的原则常彼此张力（如「快速行动」vs「谋定后动」）。**不要替作者二选一**，
那会丢掉书的精髓——原书的智慧往往恰在「何时用哪个」。

调和三步：
1. **记录冲突**：把冲突的两条原则并列写下，注明各自出处。
2. **找适用边界**：回到书里看作者各自在什么情境下主张哪条，提炼触发条件。
3. **写成判断规则**：在 `principles.md` 写「当 <情境A> → 用原则甲；当 <情境B> → 用原则乙」，
   而非删掉其中一条。

> 原则：**矛盾不是要消除的 bug，是要保留的判断维度。** 能说清「什么时候用哪个」的 skill，
> 才真正继承了原书的判断力。
