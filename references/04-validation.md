# 阶段五 · 五维验证（以"用"为验，不以"记"为验）

> **验收哲学**：验收一个由书做成的 skill，看它能否在**新场景**里做出**符合原书精神的判断**，
> 而不是看它记住了多少。记忆是结构，判断是能力。前者是地基，后者才是楼。
>
> 一本书的价值 = 它改变读者「如何判断与行动」的能力。所以五维验证里，
> **真正的验收在后三维（应用/判断/边界）**，前两维（概念/关系）只是"地基没塌"的必要前提。

---

## 一、五维验证表（每维：方法 + 标准 + 如何取证）

| 维度 | 方法 | 标准 | 如何取证 |
|------|------|------|---------|
| 概念完整 | 从 concepts.md 抽 20 个核心概念，逐一查 SKILL.md/references 是否覆盖且定义可用 | **100% 覆盖**（缺一即不过） | 列出 20 概念清单，逐条标 ✓/✗，附其落点（文件#锚点）|
| 关系完整 | 对 knowledge_graph.md 做连通性检查：每个节点至少有一条出边或入边 | **0 孤立节点** | 列出孤立节点（应为空），或贴 check_skill.py 的孤立节点计数 |
| 应用正确 | 取 cases.md 的 10 个案例，让 skill 角色扮演原书作者去处理 | 核心判断与原书**一致** | 10 个案例的 skill 应答 vs 原书答案对照表，标"一致/偏离" |
| 判断准确 | 构造 5 个**变体场景**（书中没直接讲的新情形）让 skill 推理 | 推理**符合原书精神**（不照搬条文，而是迁移原则）| 5 段推理记录 + 评语"原则迁移正确/机械套用/跑偏" |
| 边界清晰 | 构造 3 个**边界场景**（原书方法不适用的情形）问 skill | 能正确说**"不"**（识别不适用并说明理由）| 3 段应答 + 评语"正确拒绝/勉强硬答/误用" |

**前两维 vs 后三维**：
- **概念完整 / 关系完整** → 偏"**结构/记忆**"：验的是知识有没有被搬进来、连起来。机器可粗检。
- **应用正确 / 判断准确 / 边界清晰** → 偏"**判断/应用**"：验的是知识能不能用、会不会迁移、知不知边界。
- 一个能背全书却在变体场景跑偏的 skill，**不算合格**。结构满分 + 判断不及格 = 验收不通过。

**判定规则**：五维全过才算通过。任一后三维不达标，**优先回阶段二/三返工**（多半是原则没提炼到位或边界没写清），而非补记忆。

## 二、提取维度 vs 验证维度是正交的（不要合并）

阶段二的**四层**（概念 L1 / 框架 L2 / 方法 L3 / 原则 L4）是**提取维度**——
回答"从书里挖什么、挖出来放哪个文件"。

阶段五的**五维**（概念/关系/应用/判断/边界）是**验证维度**——
回答"做出来的 skill 好不好用、靠不靠谱"。

**二者不应合并成一个共享分类。** 它们方向不同：提取是"输入侧的拆解"，验证是"输出侧的检验"。
强行对齐会让人误以为"提取了原则层 = 判断维就过了"，而事实上原则被提取≠原则能被正确迁移。

**案例的双重身份**（最易混淆，单列说明）：
- 在**提取侧**：案例是每个知识要素的"**验证锚点**"，标注书中出处，用来证明这条概念/方法确实来自书、不是杜撰。
- 在**验证侧**：同一批案例是"**应用正确**"维的**素材**，用来角色扮演检验 skill 能否复现原书判断。
- 同一个案例、两种用途，**不要因为名字像就把它当成第三个分类层**。

## 三、结构验证自动化：`check_skill.py`

```bash
python3 scripts/check_skill.py <skill_dir>
```

**它检查（且只检查）这些结构项**：

- [ ] `SKILL.md` 存在，且 **< 500 行**
- [ ] `SKILL.md` 含合法 YAML frontmatter，且有 `name` / `description` 字段
- [ ] `references/` 六件套齐全：concepts / frameworks / methods / principles / cases / knowledge_graph 各一
- [ ] SKILL.md 路由表**每条路由都附具体加载指令**（形如 `references/methods.md#方法N`，"写了但调不到"判不过）
- [ ] **概念数粗检**：concepts.md 概念条目数量在合理区间（过少→提取不足）
- [ ] **孤立节点粗检**：knowledge_graph.md 中无入边也无出边的节点计数（应为 0）

**结构通过 ≠ 判断通过。**
check_skill.py 是**必要非充分**条件：它能拦住"文件缺失、超长、路由失联、图谱断裂"这类硬伤，
但**测不出**"判断对不对、边界清不清"。后三维（应用/判断/边界）**必须人工角色扮演**，无法脚本化。

> 工作流：先跑 check_skill.py 过结构门禁（红线全清），再做后三维人工验证。结构没过就别浪费人工。

## 四、变体场景测试的时机（判断准确维的特殊处理）

判断准确维考的是"书里没直接讲的新情形"，而**最真实的变体场景来自实际使用**，不是凭空编。

**建议节奏**：
1. **交付时**：先用 3-5 个自构变体场景做一轮 baseline，记录在验证日志。
2. **使用 1-2 周后**：收集 skill 实际遇到的真实新场景，挑代表性的补做一轮**回归测试**。
3. **写入日志**：把真实场景 + skill 应答 + 是否符合原书精神，**追加**进原点工作日志，形成可追溯的判断力档案。

> 真实场景暴露的偏差，往往指向阶段二原则提取的盲区——这正是 skill 自进化（道）的入口。

## 五、防错清单（汇总五轮真实书籍教训）

逐条勾选，全勾才算闭环。每条后括注它防的是哪类翻车：

- [ ] **阶段零冲突检测完成**（防：撞名覆盖已有 skill / 重复造轮子）
- [ ] **确认"文件名标 OCR ≠ Read 能读"**：扫描版必须用 PyMuPDF 验证文字层是否真存在（防：把图片当文字，垃圾进流水线）
- [ ] **阶段一输出 `book_structured.md` 已归档**（防：结构化产物丢失，后续无法回溯）
- [ ] **阶段二 Agent prompt 含验证锚点要求**（防：Agent 杜撰概念、无出处）
- [ ] **阶段二亲自读 ≥2 个核心章交叉验证**（防：全盘信 Agent，漏读/误读核心判断）
- [ ] **阶段二产出含隐含假设 + 原则冲突调和**（防：只搬明面条文，丢掉书的"言外之意"与矛盾处理）
- [ ] **skill 已存在则先评估质量**（优 → 仅增强路由；劣 → 重写。防：在烂地基上加盖）
- [ ] **SKILL.md < 500 行 + YAML 用 block scalar `|`**（防：超长拖慢加载 / frontmatter 解析失败）
- [ ] **references 六件套齐全 + 每个文件有检索提示**（防：缺件 / 有文件但 Claude 找不到该读哪份）
- [ ] **路由表每条都有具体 references 加载指令**（防："写了但调不到 = 知识不存在"）
- [ ] **五维验证全过（含案例锚点）**（防：结构达标就放行，判断/边界没测）
- [ ] **原点工作日志已追加执行记录**（防：经验不沉淀，下次重复踩坑）

## 六、OCR 质量门禁应前移到阶段一（垃圾进 = 垃圾出）

最贵的错误是**让低质 OCR 文本一路流到阶段二**：Agent 在乱码上提取，产出看似工整实则全错，
而错误要到阶段五才暴露，返工成本最高。

**门禁规则**：
- 在**阶段一**对结构化文本做可读性抽检（关键语义段是否成句、术语是否完整）。
- **可读性 < 70% → 触发重扫**（换 PyMuPDF 提嵌入层 / tesseract chi_sim 重 OCR），**不放行进阶段二**。
- 门禁前移的本质：把质量检查放在**成本最低、最易补救**的环节，而不是等到下游放大后再救。

> 同理见阶段零的 OCR 专项门禁（`references/01-preprocess.md`）：可读率 ≥60% 才进阶段二是底线，
> 阶段一这道 <70% 重扫门禁是更严的质量线。两道门禁叠加，确保"进阶段二的文本是干净的"。

## 七、🔒 溯源抽检（v2.4，应用/判断维的前置门禁）

虚构的知识再「自洽」也是污染，会让后三维**测出假合格**。故在五维之前先过溯源抽检：

1. **随机抽 5 个 L1–L4 要素**，逐条回原书核对出处真实存在且与原文一致。
2. 命中 `[待核]` 或查无实据者 → **补源或删除**后才算通过。
3. 通过后再做应用/判断/边界三维（否则是在虚构知识上做角色扮演，白测）。

防错清单补一条：
- [ ] **溯源抽检通过**：随机 5 要素回原书核对，`[待核]` 已清零（防：虚构知识混入，后三维测出假合格）
