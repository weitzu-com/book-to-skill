# 阶段三/四 · 认知架构设计 + SKILL 编写 + references 六件套

> **一句话**：把阶段二普查到的四层知识，「编册成一个可被调用的能力」——先定认知架构五要素，
> 再写 < 500 行的 SKILL.md，配齐 references 六件套，每条路由都带具体加载指令。
>
> **第一性原理**：知识入库不等于能力可用。**「写了但调不到 = 知识不存在」**。
> 本阶段的全部工夫，是让 Claude 在用户提问的那一刻，能精准地把对的知识取出来用对。

**何时检索本文**：进入阶段三/四、要设计认知架构、要写 SKILL.md、要建 references 六件套、
要写路由加载指令、或面对「已有同名 skill 该增强还是重写」的决策时。

## 1. 认知架构五要素（设计前先答这五问）

动手写文件之前，先把这五问答清楚。答错身份定位或知识分配，后面全要返工。

| 要素 | 关键问题 | 示例 |
|------|---------|------|
| 身份定位 | 这个 skill 以谁的身份说话？ | 实操者作者（村田忠嗣/Joe Girard）→ **以作者本人身份说话**；理论家学者 → 以「讲解人」身份，**不假装是作者** |
| 知识分配 | 什么放 SKILL.md，什么放 references？ | SKILL.md = 故事 + 框架 + 精选方法（< 500 行）；references = 全量知识 |
| 能力边界 | 什么能做、什么不能做？ | 明确列「不适用场景」+「可迁移场景」 |
| 行为模式 | 用户问什么 → skill 回什么？ | 先讲故事/原则，再给方法；不堆砌清单 |
| 交互协议 | 用什么命令/路由触发？ | 设计命令/路由覆盖常见查询，每条挂到具体 references |

**身份定位的判断法**：作者是否亲身实践了书中方法？
- **是**（如吉拉德卖了 13000 辆车、村田忠嗣经营过工厂）→ skill 用第一人称作者口吻，更有说服力。
- **否**（学者整理/转述他人方法）→ skill 用「讲解人」口吻，诚实标注「据某某理论」，不冒充作者。

**知识分配的红线**：SKILL.md 是「常驻内存」，每次加载都吃 token，所以只放**高频 + 入口级**内容
（招牌故事、核心框架图、最常用的几个方法名）。全量细节一律下沉 references，按需检索。

## 2. SKILL.md 铁律

| 铁律 | 要求 | 参考基准 |
|------|------|---------|
| YAML frontmatter | description 用 block scalar `\|`（多行描述不被折叠）| —— |
| 总行数 | **< 500 行**（超了就往 references 下沉）| 会计七原则 180 行 / 吉拉德 217 行 |
| references 检索提示 | 正文须告诉读者「需要 X 就去读 references/Y」| —— |
| 命令参考表 | 含命令/路由表，覆盖常见查询 | —— |

### SKILL.md 推荐骨架


## 3. references 六件套标准（缺一不可）

六个文件各管一层/一职，**每个文件顶部都要写「何时检索」提示**，让 Claude 知道何时该打开它。

| 文件 | 装什么 | 顶部「何时检索」提示示例 |
|------|--------|------------------------|
| `concepts.md` | L1 全部核心概念 + 定义 + 出处 + 锚点 | 「需要某个术语的精确定义/与近义词区别时读」 |
| `frameworks.md` | L2 框架 + 组成 + **章节对照表** | 「需要完整框架结构/各要素关系时读」 |
| `methods.md` | L3 方法 + 步骤 + 输入输出 + **验证标准** | 「用户要『怎么做』、需要可执行步骤时读」 |
| `principles.md` | L4 原则 + **边界条件 + 矛盾调和** | 「需要做价值判断/取舍、或两条原则冲突时读」 |
| `cases.md` | 10 个关键案例 + 每个配**可测试问题** | 「需要举例佐证/做应用验证/角色扮演时读」 |
| `knowledge_graph.md` | 知识图谱 + **完整度热图** + 隐含假设 | 「需要看知识全貌/查孤立节点/查隐含前提时读」 |

**逐个要点**：
- **concepts.md**：每条 = 概念名 + 一句定义 + 出处锚点；按字母或主题排序便于检索。
- **frameworks.md**：除框架本身，必须有「框架 ↔ 章节」对照表，方便溯源到原书。
- **methods.md**：每个方法四件套——步骤、输入、输出、验证标准（怎么算做对了）。
- **principles.md**：每条原则附边界条件；冲突原则写成「情境A→甲 / 情境B→乙」（见 `02-parse.md` §6）。
- **cases.md**：精选 10 个最有代表性的案例，每个配一道「可测试问题」供阶段五角色扮演验证。
- **knowledge_graph.md**：节点关系网 + 完整度热图（哪层覆盖充分、哪层稀疏）+ 全书隐含假设清单。

## 4. 路由加载指令（v2.0 核心创新）

> **原则：「写了但调不到 = 知识不存在」。** 知识写进 references 只完成一半，
> 必须让 SKILL.md 的路由表**指到具体文件 + 具体锚点**，Claude 才取得到。

格式：`<查询类型> → references/<文件>#<锚点>`，例如 `方法 → references/methods.md#方法3`。

| | 写法 | 后果 |
|---|------|------|
| 反例 ✗ | 路由表只写「方法相关看 references」 | Claude 不知打开哪个文件、定位到哪条，常答非所问或全文加载烧 token |
| 正例 ✓ | 「客户跟进话术 → references/methods.md#方法7-跟进系统」 | Claude 精准跳转，取出正确知识，token 最省 |

**自检**：写完路由表，逐条问自己「Claude 看到这条，能不能一步定位到知识？」不能则补锚点。
阶段五的 `check_skill.py` 也会扫路由加载指令是否齐全。

## 5. Skill 已存在时的决策（增强 vs 重写）

阶段零的冲突检测若发现已有同名/近似 skill，按质量决定：

| 已有 skill 质量 | 决策 | 怎么做 |
|----------------|------|--------|
| **优**（结构清晰、命令体系成熟）| **增强** | **保留**已有命令体系与正文，**只精准补路由加载指令**（把没指到锚点的路由补全），下沉缺失的 references 层。不推翻。 |
| **劣**（行数超标、无六件套、路由调不到）| **重写** | 按本文档骨架重建，旧内容择优迁移。 |

> 增强优先：成熟 skill 的命令体系是用户的肌肉记忆，推翻代价高。多数「老 skill」的真问题不是知识缺，
> 而是「写了但调不到」——补路由加载指令即可激活已有知识。
